# Portfolio 想法压力测试

## 1. 原始想法摘要

做一个面向实习投递的个人作品集网站，展示 AI 应用开发方向、CodePilot 项目、前端开发能力和工程化工作流。目标用户是技术面试官、HR 和实习招聘方。第一版做静态页面，包含 Hero、About、CodePilot、Portfolio 开发过程、Skills、Workflow、Contact 七个模块。最终希望证明 AI 应用开发表达、项目包装、React + Vite 开发、设计控制和 Claude Code 工作流能力。

## 2. 需求真实性检查

**技术面试官 / HR 真的会看作品集吗？**

- HR 大概率不会主动看作品集，除非简历上链接突出、项目经历引起兴趣。
- 技术面试官在面试前可能会打开看看，但停留时间通常不超过 2 分钟。
- 实习招聘场景下，作品集不是必须的，但有比没有强。

**作品集相比简历和 GitHub 有什么补充价值？**

- 简历是文字，GitHub 是代码，两者都不直观。
- 作品集提供一个"可直接感知"的入口，把项目包装成可浏览的内容。
- 但它不能替代简历和 GitHub，只能作为补充。

**这个项目是否只是"好看网页"，还是能证明工程能力？**

- 如果只是几个静态页面堆文字，那就是好看网页。
- 如果能展示项目决策过程、技术选型理由、迭代记录，才能证明工程能力。
- 风险在于：容易做成"看起来专业但内容空洞"的展示页。

**结论：需求真实，但价值被高估。作品集是锦上添花，不是雪中送炭。核心还是 CodePilot 本身和简历质量。**

## 3. 目标用户检查

**HR 看它时最关心什么**

- 候选人的项目经历是否真实
- 是否有可访问的在线链接
- 页面是否能快速加载、信息是否清晰
- HR 不关心技术细节，关心"这个人做过什么"

**技术面试官看它时最关心什么**

- 项目的技术深度和实现细节
- 是否有可演示的 Demo
- 代码质量（会去 GitHub 看）
- 技术选型的理由
- 遇到了什么问题、怎么解决的

**实习招聘方看它时最关心什么**

- 是否有完成度高的项目
- 是否有持续学习和迭代的证据
- 是否有工程化意识（不只是写代码）

**当前想法是否覆盖这些需求：**

- HR 需求：部分覆盖（有页面、有链接），但缺少清晰的项目摘要
- 技术面试官需求：部分覆盖（有 CodePilot 模块），但缺少技术深度展示
- 实习招聘方需求：部分覆盖（有 Workflow 模块），但"Portfolio 开发过程"对招聘方价值不大

## 4. 差异化检查

**CodePilot 是否能成为核心项目支撑**

- 如果 CodePilot 有可演示的 Demo、清晰的技术架构、可讲的难点，可以。
- 如果 CodePilot 还处于早期、功能不完整、讲不清楚内部实现，撑不起来。
- 风险：CodePilot 本身还在开发中，作品集先于项目完成，会导致内容空洞。

**Portfolio 自身是否能证明前端和工程化能力**

- 可以，但前提是 Portfolio 本身是一个有质量的项目。
- 如果只是用模板改改，或者页面很简单，证明不了什么。
- 需要在 Portfolio 本身体现组件化、响应式、性能优化等工程实践。

**Vibe Coding 工作流是否应该展示**

- 这是一个差异化点，但也是风险点。
- 如果展示得好，体现"AI 辅助开发"的工程意识。
- 如果展示得不好，会显得像是在凑内容，或者让人觉得"你只是让 AI 写代码"。
- 建议：简要提及即可，不要作为核心卖点。

**是否存在过度包装风险**

- 风险很高。
- 如果把 Portfolio 包装成"AI 驱动的作品集"，但实际只是静态页面，就是过度包装。
- 如果把 Vibe Coding 包装成"革命性工作流"，但实际只是用 Claude Code 写代码，也是过度包装。
- 原则：如实展示，不要夸大。

## 5. MVP 收窄检查

当前 MVP：Hero、About、CodePilot、Portfolio 开发过程、Skills、Workflow、Contact

**判断：MVP 过大。**

### 必须保留

| 模块 | 理由 |
|------|------|
| Hero | 首屏必须有，但只需要一句话 + 视觉基调 |
| About | 简短个人介绍，不能省略 |
| CodePilot | 核心项目展示，是作品集存在的理由 |
| Contact | 投递必需，联系方式不能省 |

### 可以延后

| 模块 | 理由 |
|------|------|
| Portfolio 开发过程 | 第一版不需要展示"怎么做作品集"，先有作品集再说 |
| Skills | 技能列表在简历上已有，作品集里加不加价值不大 |
| Workflow | Vibe Coding 工作流可以后续版本再展示，第一版优先把项目讲清楚 |

### 明确不做

| 模块 | 理由 |
|------|------|
| 复杂动效 | 时间成本高，对投递价值低 |
| 后端功能 | 静态页面足够，不需要 API |
| AI 问答 | 第一版不需要，过度设计 |
| 多语言 | 不需要，中文即可 |

## 6. 技术可控性检查

**React + Vite + TypeScript + Tailwind 是否合适**

- 合适。这是当前主流前端技术栈，学习成本可控，适合静态页面项目。
- 但要注意：如果对 React 不够熟悉，可能会在组件设计上花太多时间。
- 替代方案：如果 React 熟练度不够，可以用纯 HTML + CSS + 少量 JS，先把内容做出来。

**是否需要后端**

- 不需要。静态页面 + GitHub Pages / Vercel 部署即可。

**是否需要 AI 问答**

- 不需要。第一版不需要任何 AI 功能，作品集本身是展示，不是产品。

**是否需要数据库**

- 不需要。所有内容写在代码里或用 JSON 文件。

**是否需要登录**

- 不需要。

**是否需要复杂动效**

- 不需要。简单的滚动动画、hover 效果即可。
- 复杂动效（如 3D、粒子、页面转场）时间成本高，对投递价值低。

**结论：技术栈合理，但要严格控制范围，不要引入不必要的复杂度。**

## 7. 成本与时间检查

**是否适合短期完成**

- 如果 MVP 收窄到 4 个模块（Hero、About、CodePilot、Contact），可以在 1-2 周内完成。
- 如果保持 7 个模块，可能需要 3-4 周，容易拖延。

**是否适合边做边投实习**

- 不建议。先完成最小版本，上线后再投递。
- 边做边投容易导致两个都做不好。

**哪些内容最容易拖慢进度**

- 设计基调确定（容易反复修改）
- CodePilot 模块内容组织（需要先整理清楚项目本身）
- 响应式适配（不同设备的布局调整）
- 动效实现（容易陷入细节）

**哪些功能应该后置**

- Portfolio 开发过程展示
- Skills 模块
- Workflow 模块
- 任何动效优化
- 多端适配细节

## 8. 简历与面试价值检查

**这个项目写进简历是否有价值**

- 有价值，但需要正确表述。

**CodePilot 是核心 AI 项目**

- CodePilot 应该作为独立项目写进简历，展示 AI 应用开发能力。
- 作品集只是 CodePilot 的一个展示入口，不是 CodePilot 本身。

**Portfolio 是作品集展示和前端工程项目**

- Portfolio 可以写进简历，但应该表述为"个人作品集网站"，展示前端开发和项目包装能力。
- 不要把 Portfolio 包装成"AI 驱动的智能作品集"，这是虚假包装。

**不要把 Portfolio 包装成虚假的 AI Agent 项目**

- Portfolio 本身不是 AI 项目，它只是一个展示页面。
- 如果在 Portfolio 里集成了 AI 功能（如聊天机器人），那才是 AI 项目，但第一版不需要。
- 如实描述：用 React + Vite + Tailwind 构建的静态作品集网站。

## 9. 主要风险

**风险 1：页面好看但内容空**
- 表现：视觉设计不错，但 CodePilot 讲不清楚，项目经历描述空洞。
- 规避：先整理 CodePilot 的技术架构、核心功能、遇到的难点，再开始做页面。

**风险 2：CodePilot 讲不清**
- 表现：面试时被问到 CodePilot 的实现细节，答不上来。
- 规避：在做作品集之前，先写一份 CodePilot 的技术说明文档，梳理清楚架构和亮点。

**风险 3：项目过度包装**
- 表现：把静态页面包装成"AI 驱动"，把用 Claude Code 写代码包装成"革命性工作流"。
- 规避：如实描述，用词准确，不要夸大。

**风险 4：动效喧宾夺主**
- 表现：花大量时间做动画效果，但内容没准备好。
- 规避：第一版不做复杂动效，先完成内容，再考虑视觉优化。

**风险 5：花太多时间在视觉细节**
- 表现：反复调整颜色、间距、字体，迟迟不定稿。
- 规避：先确定设计基调，然后严格执行，不要反复修改。

**风险 6：没有上线**
- 表现：做了很久但没有部署，投递时没有可访问的链接。
- 规避：第一版用最简单的部署方式（GitHub Pages 或 Vercel），先上线再说。

**风险 7：没有真实链接**
- 表现：作品集里放了项目链接，但打不开或者不是真实项目。
- 规避：确保 CodePilot 有可访问的 Demo 或 GitHub 仓库。

**风险 8：没有可讲的技术难点**
- 表现：面试时被问"这个项目有什么技术难点"，答不上来。
- 规避：在做作品集时，刻意记录技术决策和解决方案，形成可讲的故事。

## 10. 最终判断结论

**结论：可以做，但需要收窄范围。**

理由：
1. 作品集对实习投递有辅助价值，但不是决定性因素。
2. 当前 MVP 范围过大，需要收窄到 4 个核心模块。
3. CodePilot 是核心，作品集是载体，不要本末倒置。
4. 需要先确保 CodePilot 本身有足够内容可讲，再开始做作品集。
5. 技术栈合理，但要严格控制复杂度。

## 11. 建议后的 MVP 范围

### 当前第一版做什么

| 模块 | 内容 | 深度 |
|------|------|------|
| Hero | 一句话介绍 + 视觉基调 | 简洁 |
| About | 简短个人介绍、技术方向、联系方式 | 精炼 |
| CodePilot | 项目介绍、核心功能、技术架构、Demo 链接 | 深入 |
| Contact | 邮箱、GitHub、LinkedIn（如有） | 最简 |

### 当前第一版不做什么

- Portfolio 开发过程展示
- Skills 模块
- Workflow 模块
- 复杂动效
- 后端功能
- AI 问答
- 多语言
- 深色/浅色主题切换

### 后续版本再做什么

- Portfolio 开发过程展示（体现工程化能力）
- Skills 模块（如果内容足够丰富）
- Workflow 模块（如果 Vibe Coding 工作流有独特价值）
- 简单动效（滚动动画、hover 效果）
- 响应式优化（移动端适配）

**允许进入 Phase 1-3：输出想法判断结论。**

原因：
1. 原始想法方向正确，适合实习投递场景。
2. MVP 已收窄到 4 个核心模块，范围可控。
3. 技术栈合理，没有过度设计。
4. 主要风险已识别，有明确的规避方式。
5. 需要先输出想法判断结论，确认方向后再进入 PRD 阶段。
