# Portfolio PRD 追问对齐清单

## 1. 当前 PRD 初步判断

初版 PRD 已收窄到 4 个模块（Hero、About、CodePilot、Contact），与压力测试和判断结论一致，方向没有跑偏。

但仍存在以下风险需要用户确认：

- **Hero 文案未定：** PRD 写了"2027 届人工智能本科生"和"AI 应用开发 / Agent 开发 / Python 后端"，但这些信息未经用户确认，可能是 AI 自行推断。
- **CodePilot 展示深度未定：** PRD 列了"项目定位、解决的问题、核心流程、技术栈、当前进展、可讲的技术点"，但哪些信息是真实的、哪些是占位的，用户没有确认。
- **联系方式未确认：** Email、GitHub 账号、简历 PDF 是否存在，全部未确认。
- **语言选择未定：** PRD 全文中文，但技术关键词是英文，没有确认最终语言方案。
- **视觉方向未定：** PRD 没有涉及视觉风格，但压力测试提到了"暗色高级"和"极简"两个方向。
- **上线和时间未定：** 部署平台、是否必须上线、期望完成时间，全部未确认。

**结论：初版 PRD 框架合理，但内容层面有多个关键信息需要用户确认后才能固化为最终 PRD。**

## 2. 必须回答的问题

以下问题必须由用户确认，不能由 AI 自己决定。

### 2.1 Hero 主文案

追问：

- Hero 主标题用中文还是英文？
- 是否要出现"AI 应用开发 / Agent 开发 / Python 后端"三个方向？
- Hero 是否只突出 CodePilot，还是同时突出个人身份？
- 是否要写"2027 届人工智能本科生"？

候选方向（待用户确认）：

**方向 A：身份 + 方向 + 项目**
> 2027 届人工智能本科生 / 聚焦 AI 应用开发与 Agent 开发 / 当前核心项目：CodePilot

**方向 B：一句话定位 + 项目**
> 做 AI 应用的开发者，当前在做 CodePilot

**方向 C：纯英文技术向**
> AI Application Developer | Building CodePilot

以上均为候选，具体措辞待用户确认方向后再定。

### 2.2 CodePilot 展示深度

追问：

- CodePilot 展示到什么深度？
- 是否展示技术栈？（如 Python / LangChain / FastAPI 等）
- 是否展示架构流程？（如用户输入 → LLM 处理 → 输出）
- 是否展示版本迭代？（如 v1 做了什么、v2 做了什么）
- 是否展示真实 GitHub 链接？
- 是否展示 Demo 链接？
- 当前哪些信息是真实可讲的？

**明确要求：如果信息未确认，统一写 TBD，不允许编造。**

需要用户回答的核心问题：
1. CodePilot 目前是什么状态？（开发中 / 已完成 / 已上线）
2. CodePilot 的 GitHub 仓库是否存在？链接是什么？
3. CodePilot 有没有可访问的 Demo？
4. CodePilot 的技术栈是什么？
5. CodePilot 面试时能讲的技术难点是什么？

### 2.3 简历入口

追问：

- 当前是否已有真实简历 PDF？
- 是否第一版就放简历下载按钮？
- 如果没有 PDF，是否只放"Resume: TBD"？
- 是否避免空链接？

**PRD 当前处理方式：** 如果简历 PDF 不存在，先写 TBD，不做下载按钮。需要用户确认这个处理方式是否接受。

### 2.4 Contact 信息

追问：

- Contact 第一版只放 Email + GitHub，还是加其他平台？
- Email 是否使用用户确认后的邮箱？
- GitHub 是否使用当前仓库账号（Strange-Men）？
- 是否需要 LinkedIn / BOSS / 牛客等入口？

**不要编造任何联系方式。** 需要用户提供：
1. 用于展示的 Email 地址
2. GitHub 用户名（是否就是 Strange-Men）
3. 是否需要其他平台入口

### 2.5 语言选择

追问：

- 第一版用中文、英文，还是中英混合？
- 面向国内实习投递，是否中文为主更合适？
- 技术标题是否保留英文？（如 CodePilot、Hero、About）

**候选方案（待用户确认）：**

| 方案 | 说明 |
|------|------|
| A：纯中文 | 所有文案中文，技术名词保留英文（如 CodePilot、GitHub） |
| B：纯英文 | 全部英文，适合外企或海外投递 |
| C：中英混合 | 主体中文，技术标签和标题用英文 |

### 2.6 视觉方向

只追问，不做设计。

问题：

- 第一版视觉是极简白底，还是暗色高级？
- 是否参考之前的 Dark Technical Editorial 方向？
- 是否先静态视觉，后续再做动效？
- 是否完全禁止复杂动效进入 MVP？

**注意：本轮不做设计，只确认方向。** 设计基调在九准备第 7 步才做。

### 2.7 上线与链接

追问：

- 第一版是否必须上线？
- 部署平台倾向 Vercel 还是 GitHub Pages？
- CodePilot 是否已有可访问仓库链接？
- 如果没有 Demo，是否只放 GitHub？

### 2.8 投递时间约束

追问：

- 这个 Portfolio 是否服务于近期实习投递？
- 是否需要优先"能上线、能讲清"而不是"视觉极致"？
- 第一版期望几天内完成？

## 3. PRD 中可能需要删除或收紧的内容

逐条检查初版 PRD 中的风险点：

### 3.1 Contact 中出现还不存在的简历下载

- **是否存在风险：** 是。PRD 写了"简历入口（如果简历 PDF 不存在，先写 TBD，不做下载按钮）"，但没有确认简历是否真实存在。
- **建议处理：** 用户确认简历 PDF 状态。如果不存在，Contact 中不出现简历相关文案，避免"TBD"给人不专业的印象。

### 3.2 CodePilot 展示内容过深导致页面复杂

- **是否存在风险：** 是。PRD 列了"项目定位、解决的问题、用户输入、系统输出、核心流程、技术栈概述、当前进展、可讲的技术点"共 8 项内容，如果每一项都展开，CodePilot 模块会变得很长。
- **建议处理：** 用户确认展示深度。如果信息不足，先做简洁版（定位 + 核心功能 + 技术栈 + 链接），后续再补充。

### 3.3 MVP 被扩成完整个人主页

- **是否存在风险：** 低。PRD 已明确只做 4 个模块，压力测试也确认了范围。
- **建议处理：** 保持当前 4 模块范围，不在开发过程中增加新模块。

### 3.4 视觉和动效抢走 CodePilot 的重点

- **是否存在风险：** 中。PRD 没有涉及视觉和动效，但后续设计阶段可能会引入。
- **建议处理：** 在最终 PRD 中明确"视觉和动效只能服务内容，不能反客为主"。

### 3.5 Portfolio 被包装成核心 AI 项目

- **是否存在风险：** 低。PRD 已明确"Portfolio 是辅助展示载体，不是核心 AI 项目"。
- **建议处理：** 保持当前定位，不在后续阶段改变。

### 3.6 出现未确认链接、数据、经历

- **是否存在风险：** 是。PRD 中的 CodePilot 信息、联系方式、简历入口都未确认。
- **建议处理：** 所有未确认信息统一写 TBD，用户确认后再替换。不允许编造。
