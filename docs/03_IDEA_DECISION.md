# Portfolio 想法判断结论

## 1. 判断结果

**可以做，但需要收窄范围。**

## 2. 为什么不是"直接值得做"

Portfolio 对实习投递有帮助，但不是核心竞争力本身。

- 作品集的价值取决于 CodePilot 是否讲清楚。如果 CodePilot 本身内容空洞，作品集做得再好看也没用。
- 如果只做成好看的网页，价值有限。招聘方不会因为页面漂亮就发 offer。
- 如果过度包装，会削弱可信度。把静态页面包装成"AI 驱动"、把用 Claude Code 写代码包装成"革命性工作流"，面试时会被追问，反而减分。

**结论：作品集是锦上添花，不是雪中送炭。核心还是 CodePilot 本身和简历质量。**

## 3. 为什么仍然值得做

尽管不是核心竞争力，Portfolio 仍然有明确价值：

- **它可以作为简历和 GitHub 的补充入口。** 简历是文字，GitHub 是代码，作品集提供一个"可直接感知"的展示入口。
- **它可以集中展示 CodePilot。** 把项目包装成可浏览的内容，比让招聘方自己去翻 GitHub 更高效。
- **它可以证明基础前端实现能力。** React + Vite + Tailwind 技术栈的实践。
- **它可以证明工程化、小步迭代、版本控制和上线能力。** Portfolio 本身的开发过程就是能力证明。
- **它适合短期完成并投入实习投递。** 收窄范围后，1-2 周可以完成。

## 4. 最终项目定位

Portfolio 是一个面向实习投递的静态个人作品集网站，核心目标是让招聘方快速理解我的 AI 应用开发方向、CodePilot 项目能力和基础前端工程能力。

## 5. 核心价值排序

1. **讲清楚 CodePilot** — 作品集存在的根本理由
2. **说明我的求职方向** — 让招聘方快速判断匹配度
3. **展示基础前端工程能力** — 技术栈实践证明
4. **展示可上线、可维护、可迭代的开发流程** — 工程化意识证明
5. **提供联系方式和跳转入口** — 投递必需

**明确约束：视觉和动效只能服务内容，不能反客为主。**

## 6. 第一版 MVP 范围

第一版只做 4 个模块：

| 模块 | 作用 |
|------|------|
| **Hero** | 让用户 5 秒内知道我是谁、做什么方向 |
| **About** | 简短说明我的背景、求职方向和技术关注点 |
| **CodePilot** | 作为核心项目深度展示 |
| **Contact** | 提供 GitHub / Email / 简历入口，如果简历文件不存在则先不写下载按钮 |

## 7. 第一版明确不做

以下内容第一版不做：

- AI 问答
- 后端
- 登录
- 数据库
- 博客系统
- 多项目堆叠
- 复杂动效
- ReactBits
- WebGL
- cursor spotlight
- Portfolio 开发过程展示
- Skills 大而全展示
- Workflow 详细展示

**说明：这些内容不是永久不做，而是第一版先不做。** 先完成核心内容，再考虑扩展。

## 8. 后续版本可以做

第一版 MVP 完成后，后续可以考虑：

- Portfolio 开发过程（体现工程化能力）
- Skills 简洁展示（如果内容足够丰富）
- Workflow 小步迭代展示（如果 Vibe Coding 工作流有独特价值）
- 响应式增强（移动端适配）
- 轻量动效（滚动动画、hover 效果）
- 上线部署（GitHub Pages 或 Vercel）
- 简历 PDF 入口

**前提：必须等第一版 MVP 完成后再评估。**

## 9. 最大风险和约束

| 风险 | 约束策略 |
|------|----------|
| CodePilot 内容讲不清，Portfolio 就会变空 | 先整理 CodePilot 的技术架构、核心功能、遇到的难点，再开始做页面 |
| 过度追求视觉会浪费时间 | 第一版不做复杂动效，先完成内容，再考虑视觉优化 |
| 过度包装会降低可信度 | 如实描述，用词准确，不要夸大 |
| 没有上线链接会削弱展示价值 | 第一版用最简单的部署方式，先上线再说 |
| 项目范围扩大后会影响实习投递节奏 | 严格控制 MVP 范围，不要在开发过程中增加新模块 |

## 10. 是否允许进入 PRD 阶段

**允许进入 Phase 2-1：基于判断结论生成初版 PRD。**

前提：PRD 必须围绕收窄后的 MVP（Hero、About、CodePilot、Contact），不允许重新扩大范围。
